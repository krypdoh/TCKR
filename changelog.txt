TCKR Changelog - March 8, 2026
v1.0.2026.0308.1005

  Changes (2026-03-08):
  - TL;DR: Numpy enhancements and timing fixes. Now syncs to display's refresh rate resulting in perfectly smooth
    scrolling.
  - Performance optimizations — batch 1 (low-risk, no behavioral changes):
    * Moved `import numpy as np` to module level — eliminates repeated import lookups on every price update tick.
    * Replaced list-build + `np.array()` pattern in `_process_price_update_deferred` with preallocated
      `np.empty((n, 2), dtype=np.float32)` filled in-place — removes intermediate Python list allocations
      and the extra copy performed by `np.array()`.
    * Lowered Numba batch-processing threshold from `> 5` to `> 0` — small portfolios (including the
      default 3-index setup) now always hit the Numba path instead of the pure-Python fallback.
    * Lowered parallel glow-detection threshold from `> 10` to `> 0` — `parallel_glow_effect_detection`
      (prange) is now used for all non-empty price sets when Numba is available.
    * Changed `np.array(list)` → `np.ascontiguousarray(list, dtype=np.float32)` for parallel glow inputs,
      guaranteeing C-contiguous memory layout for optimal Numba JIT performance.
    * Removed 3 redundant inline `import numpy as np` statements inside hot paths.
    * Updated fallback conditions (`<= 5` → `<= 0`, `<= 10` → `<= 0`) to match the new thresholds.

  - Anti-judder scroll fixes — batch 2:
    * Removed `time.sleep()` from `TickerGLWidget.paintEvent` — sleeping inside Qt's paint callback
      blocks the entire event loop, causing pending timers and signals to queue up and fire in bursts
      when sleep returns, which is the primary source of periodic scroll judder on Windows.
    * Replaced sleep-based frame pacing with a `QTimer(PreciseTimer)` at 15 ms driving `self.update()`.
      The timer fires from the event loop so the loop stays free every frame. The display's VSync /
      DWM composition naturally caps the swap rate to the monitor refresh rate.
    * Removed redundant `import time` inside `paintEvent`; uses `self.ticker_window._time_module`
      (already cached in `__init__`) instead.
    * Glass glare effect (`apply_glass_glare_effect`) now caches to a `QPixmap` on first draw and blits
      with a single `drawPixmap` call each frame. Previously ~20 painter operations (gradient allocations,
      polygon fills, fillRect loops) ran at 60 Hz for a fully static visual layer. Cache is invalidated
      only when window dimensions change (essentially never while docked). Visual output is pixel-identical
      to the previous implementation (Plus composition mode is preserved on the cache painter so overlapping
      layers stack additively, exactly matching the original direct-Plus-on-widget behaviour).
    * Eliminated wasted Python `cycle_positions` list build in `paint_ticker` when `USE_OPT=True` —
      the list was constructed every frame then immediately discarded in favour of the Numba array.
      Moved it inside the `else:` fallback block where it is actually used.

  - True VSync lock via DwmFlush — batch 3:
    * Added `_VSyncThread` class: a `QThread` that calls `DwmFlush()` in a background thread once per
      vertical sync interval and emits a `vsync` signal to trigger `update()` on the main thread via a
      queued connection.
    * Replaced the `QTimer`-based render driver (16 ms / 62.5 FPS) with `_VSyncThread`. The timer
      approach had a residual beat frequency: at 62.5 FPS on a 60 Hz display, ~2–3 times per second a
      timer tick would land inside an already-occupied VSync window, holding that frame for ~33 ms
      instead of ~16 ms (single-frame hitch). `DwmFlush()` wakes at the exact hardware VSync boundary
      so every frame maps 1:1 to a display swap — zero beat frequency, zero hitching.
    * Frame rate is now auto-adapted to the monitor's native refresh rate: 60 Hz displays render at
      60.0 FPS, 120 Hz at 120.0 FPS, 144 Hz at 144.0 FPS, etc. Scroll speed is unaffected because
      movement uses time-based delta (`speed * delta_time * 60.0`) not frame-count math.
    * `DwmFlush()` blocks inside the background thread, so the Qt event loop stays completely free
      between frames — timers, WebSocket data, and UI events all process normally.
    * Fallback: if DWM is unavailable (non-Windows or DWM disabled), the thread sleeps 16 ms per
      iteration, equivalent to the previous QTimer approach.

  - DWM post-lock/unlock throttle recovery — batch 4:
    * When Windows is locked, DWM enters a low-power composition mode and `DwmFlush()` returns at the
      lock screen's reduced rate (observed ~42 FPS / 23.8 ms). After unlock, DWM can stay stuck at this
      throttled rate indefinitely, causing TCKR to report ~42 FPS with visible judder until restart.
    * Added self-healing logic to `_VSyncThread.run()`: tracks consecutive frames where `DwmFlush()`
      takes >20 ms (i.e. <50 Hz). After 10 consecutive slow frames, the thread sleeps 500 ms to allow
      DWM to reinitialise at the monitor's native rate, then resumes normal operation.
    * A 30-second cooldown between recovery attempts prevents retrigger loops while still firing once
      per lock/unlock cycle.
    * Recovery is logged to console: `[PERF] DwmFlush throttled after lock/unlock — pausing 500 ms
      for DWM to stabilise`

TCKR Changelog - January 21, 2026
v1.0.2026.0121.1358

    Changes (2026-01-21):
    - WebSocket subscription robustness:
      * Added `queued_subscriptions` to buffer subscribe requests when the socket exists but isn't connected.
      * Flushed queued subscriptions from the websocket `on_open` handler so subscriptions are reliably sent from the connection thread.
      * `subscribe_symbols` now enqueues symbols when not connected (avoids "socket is already closed" errors) and attempts `connect()` if no socket exists.
      * When connected, `subscribe_symbols` computes a diff and sends only required `subscribe` / `unsubscribe` messages (reduces churn and network traffic).
      * Enhanced send error logging for per-symbol failures (helps diagnose closed/failed sockets).
      * Short Qt single-shot retry remains for transient startup races; overall flow now uses an explicit queue + flush on open.

    - Incremental pixmap updates & throttling:
      * Implemented per-symbol incremental pixmap rebuilds to avoid full rebuilds on each price change.
      * Added per-`TickerWindow` pending queue + dedupe set and an incremental rebuild `QTimer` that processes a small batch per tick.
      * Tunable settings: `websocket_incremental_interval_ms`, `websocket_incremental_max_per_tick` to balance smoothness vs latency.
      * `queue_incremental_pixmap_updates(...)` is used to schedule work and `build_ticker_pixmaps_for_symbols(...)` performs the partial rebuilds.

    - Secondary ticker data sync fix:
      * Fixed `N/A` price displays in secondary ticker windows by normalizing `ticker.stocks` (`[s[0] for s in load_stocks()]`) before assigning `ticker.prices`.
      * Ensures keys/ordering match across tickers so incremental updates and lookups return values instead of `None`.

    - Misc / CI:
      * Ran static syntax checks after edits — no syntax errors found.

    Notes:
    - These changes address startup subscription races and eliminate noisy "socket is already closed" errors seen when subscribing before the connection handshake completes.
    - The incremental pixmap approach drastically reduces render-time stutter by limiting expensive rebuilds to only affected symbols and by throttling work across frames.
    - WebSocket Zero-Interruption Mode: Completely eliminated jitter from real-time price updates
      * Removed batch processing timer that was causing micro-interruptions every 150ms
      * WebSocket messages now silently accumulate in buffer with zero CPU impact during scrolling
      * Single 10-second timer processes entire buffer + rebuilds visuals in one operation
      * VeryCoarseTimer priority ensures processing only happens during idle time
      * Result: Butter-smooth 60 FPS scrolling with price updates every 10 seconds
      * Trade-off: 10-second update latency accepted for maximum smoothness
    - WebSocket Message Batching & Throttling: Optimized real-time data processing
      * Configurable batch interval (default 150ms) to group incoming messages
      * Per-symbol rate limiting (default 100ms minimum between updates)
      * Change threshold filtering (default 0.1% minimum price change required)
      * Reduces unnecessary visual rebuilds while maintaining price accuracy
      * Settings: websocket_batch_interval, websocket_min_update_interval, websocket_change_threshold
    - WebSocket Thread Safety: Fixed QTimer threading issues
      * Moved QTimer creation from WebSocket thread to Qt main thread
      * Prevents "Timers can only be used with threads started with QThread" errors
      * WebSocket now only populates buffer, main thread handles all Qt operations
